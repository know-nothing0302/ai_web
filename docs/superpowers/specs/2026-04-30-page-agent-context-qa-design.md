# Page Agent 页面上下文问答设计

## 目标

本次设计聚焦为 `ai_web` 集成第一阶段 `page agent` 能力，满足以下目标：

1. 在全站主要页面提供统一的问答入口
2. 回答用户问题时优先结合当前页面内容
3. 在非详情页或当前页面信息不足时，支持检索站内内容并返回原文链接
4. 使用 `DeepSeek` 作为页面问答主模型链路
5. 保持最小实现范围，不引入自动点击、自动填表或自动发布

## 范围

### 本次包含

- 全站统一展示 `page agent` 问答入口
- 页面上下文采集与结构化上传
- 后端统一问答编排接口
- 基于当前页面内容的优先回答
- 基于站内文章内容的补充检索
- 回答中返回站内文章链接与原文链接
- 关键链路日志，便于排障与效果追踪

### 本次不包含

- 自动点击、自动填写、自动跳转
- 代替用户执行发布、删除、保存等操作
- 向量数据库与语义检索
- 跨页面长期记忆
- 对外开放的独立接口文档

## 背景

当前系统已具备以下基础：

- 前端已实现资讯列表、文章详情、订阅页、内容中枢等主要页面
- 后端已有 AI 能力接入与文章内容服务
- 环境配置已支持 `DEEPSEEK_API_BASE_URL` 与 `DEEPSEEK_MODEL`
- 后端已有一个 `page-agent` 兼容接口雏形，但该接口当前仅转发用户最后一条消息，尚未真正消费当前页面上下文

因此，第一阶段不直接建设完整 GUI Agent，而是建设“页面上下文问答助手”，优先解决“这页讲什么”“这段是什么意思”“站内还有哪些相关文章”这类高频问题。

## 用户场景

### 文章详情页

- “这篇文章的核心观点是什么？”
- “这段话是什么意思？”
- “这篇文章适合哪些人阅读？”
- “站内还有没有类似文章？”

### 资讯列表页

- “当前页主要是什么内容？”
- “这个栏目和另一个栏目有什么区别？”
- “帮我找站内关于医学 AI 的文章，并给我链接”

### 订阅页

- “每日、每周、即时推送有什么区别？”
- “我现在这个页面是在配置什么？”

### 内容中枢页

- “自动提取和 AI 提炼有什么区别？”
- “保存草稿和一键发布分别适合什么场景？”
- “系统里有没有类似主题的已发布文章？”

## 设计原则

- 优先解释，不执行操作
- 优先基于当前页面回答，不足时再补站内检索
- 只上传结构化页面信息，不上传整页原始 DOM
- 仅传输用户当前可见且本就可访问的信息
- 回答必须可追溯，检索型回答要附来源链接
- 保持最小改动，复用现有文章数据与登录态

## 总体方案

### 前端

- 在全站主要页面挂载统一 `page agent` 入口
- 用户发起提问时，前端构造结构化 `page context`
- 前端将 `用户问题 + 页面上下文` 提交给后端问答接口
- 前端展示回答、引用来源与链接

### 后端

- 新增统一页面问答编排接口
- 根据 `pageType` 识别页面类型和可用上下文
- 若当前页面上下文足够，则直接组织模型回答
- 若当前页面上下文不足，则触发站内检索，补充相关文章信息
- 调用 `DeepSeek` 生成最终回答
- 返回答案、来源链接与本次是否触发检索的元信息

### 模型

- `DeepSeek` 作为页面问答主模型
- 提示词中明确要求优先使用当前页面信息
- 提示词中明确禁止编造站内不存在的文章、链接、状态或页面信息

## 页面上下文设计

### 通用字段

所有页面统一上传以下基础字段：

```json
{
  "question": "这页主要讲什么？",
  "pageType": "article_list",
  "route": "/",
  "pageTitle": "资讯发现",
  "selectionText": "",
  "context": {}
}
```

字段说明：

- `question`：用户输入的问题
- `pageType`：页面类型，用于后端分流
- `route`：当前路由
- `pageTitle`：页面标题或页面主标题
- `selectionText`：用户主动选中的文本，若存在则优先解释
- `context`：页面类型对应的结构化上下文

### 文章详情页

```json
{
  "pageType": "article_detail",
  "route": "/articles/:id",
  "pageTitle": "文章详情",
  "selectionText": "",
  "context": {
    "articleId": "uuid",
    "title": "文章标题",
    "summary": "文章摘要",
    "contentPreview": "正文截断内容",
    "author": "作者",
    "publishedAt": "2026-04-30T08:00:00.000Z",
    "channelCode": "medical-frontier",
    "channelName": "医学前沿",
    "originalUrl": "https://example.com"
  }
}
```

要求：

- `contentPreview` 只传截断后的正文片段
- 若用户选中部分正文，则 `selectionText` 优先级高于 `contentPreview`

### 资讯列表页

```json
{
  "pageType": "article_list",
  "route": "/",
  "pageTitle": "资讯发现",
  "selectionText": "",
  "context": {
    "keyword": "医学AI",
    "category": "",
    "channelCode": "medical-frontier",
    "channelName": "医学前沿",
    "currentPage": 1,
    "pageSize": 9,
    "items": [
      {
        "id": "uuid",
        "title": "文章标题",
        "summary": "文章摘要",
        "author": "作者",
        "publishedAt": "2026-04-30T08:00:00.000Z"
      }
    ]
  }
}
```

要求：

- `items` 只传当前页结果
- 不传完整正文
- 回答“当前页是什么内容”时应优先基于当前筛选条件和当前结果摘要

### 订阅页

```json
{
  "pageType": "subscription",
  "route": "/subscription",
  "pageTitle": "智能订阅",
  "selectionText": "",
  "context": {
    "enabled": true,
    "frequency": "daily",
    "channelCodes": ["medical-frontier", "edu-plus-ai"]
  }
}
```

### 内容中枢页

```json
{
  "pageType": "admin",
  "route": "/admin",
  "pageTitle": "内容中枢",
  "selectionText": "",
  "context": {
    "extractUrl": "https://example.com/article",
    "titleDraft": "草稿标题",
    "summaryDraft": "草稿摘要",
    "channelCode": "medical-frontier",
    "publishedAt": "2026-04-30T08:00:00.000Z",
    "filterChannel": "",
    "selectedCount": 0,
    "articleListPreview": [
      {
        "id": "uuid",
        "title": "文章标题",
        "status": "published",
        "channelCode": "medical-frontier"
      }
    ]
  }
}
```

要求：

- 仅传页面中当前已可见的草稿与列表摘要
- 不传超长正文全文
- 不传隐藏字段或未展示的后台敏感信息

## 后端接口设计

建议新增业务接口：

- `POST /api/page-agent/qa`

请求体：

```json
{
  "question": "帮我概括当前页面内容",
  "pageType": "article_detail",
  "route": "/articles/123",
  "pageTitle": "文章详情",
  "selectionText": "",
  "context": {
    "articleId": "123",
    "title": "文章标题"
  }
}
```

响应体：

```json
{
  "answer": "当前文章主要介绍了……",
  "sources": [
    {
      "type": "current_page",
      "title": "当前页面",
      "url": "/articles/123"
    },
    {
      "type": "article",
      "articleId": "456",
      "title": "相关文章标题",
      "url": "/articles/456",
      "originalUrl": "https://example.com/source"
    }
  ],
  "meta": {
    "usedCurrentPage": true,
    "usedSiteSearch": true,
    "model": "deepseek-v4-flash"
  }
}
```

### 字段约束

- `answer`：最终给用户的回答
- `sources`：当前回答引用的来源
- `meta.usedCurrentPage`：是否使用了当前页面上下文
- `meta.usedSiteSearch`：是否触发了站内检索
- `meta.model`：使用的模型名称

## 页面分流与问答编排

后端按以下顺序编排：

1. 校验登录态与请求结构
2. 基于 `pageType` 判断当前页面可直接回答的范围
3. 优先使用 `selectionText`
4. 若当前页面信息足够，则直接调用模型回答
5. 若当前页面信息不足，或用户问题明显要求站内延伸检索，则执行站内检索
6. 将 `页面上下文 + 检索结果摘要 + 用户问题` 一起发给模型
7. 返回回答和引用来源

### 触发站内检索的典型问题

- “站内还有哪些相关文章？”
- “帮我找最近发布的相关内容”
- “有没有关于这个主题的其他文章？”
- “系统里是否已有类似内容？”

### 不触发站内检索的典型问题

- “这篇文章讲什么？”
- “这段话是什么意思？”
- “当前这个页面是在配置什么？”

## 站内检索设计

第一阶段仅采用轻量检索，不引入向量库。

### 检索范围

- 已发布文章优先
- 检索字段优先包含：
  - `title`
  - `summary`
  - `content`
  - `category`
  - `channelCode`

### 检索策略

- 根据用户问题提取关键词
- 合并当前页面上下文中的栏目、分类、标题关键词
- 优先召回当前主题强相关内容
- 默认返回 `1-3` 条结果
- 每条结果返回：
  - 站内文章标题
  - 站内详情页链接
  - 原文链接
  - 简短摘要

## 提示词策略

系统提示词约束：

- 你是 `AI徐医` 站内页面问答助手
- 优先基于当前页面信息回答
- 当前页面不足时，才可使用站内检索结果补充
- 不得编造站内不存在的文章、链接、页面状态、用户配置或后台数据
- 若信息不足，应明确说明“当前页面和站内结果无法确认”
- 回答应尽量简洁、清楚，适合教师、学生和管理人员理解

输出要求：

- 优先直接回答问题
- 检索型问题需提供可点击的站内链接
- 若涉及来源，应清楚区分“当前页面信息”和“站内补充信息”

## 前端交互设计

### 入口

- 全站主要页面显示统一入口
- 入口样式保持一致，不按页面拆分不同入口

### 面板

- 支持输入问题并查看回答
- 支持展示来源链接
- 支持在回答中展示“当前页面回答”与“站内检索补充”的区别

### 展示规则

- 单纯解释类问题，仅展示回答
- 检索类问题，回答下方展示来源卡片或链接列表
- 若无可用来源，则明确提示无法确认

## 安全与边界

- 不执行任何页面操作
- 不调用发布、删除、保存等写接口
- 不将用户不可见数据作为模型上下文
- 内容中枢页仅传输当前已显示的草稿与列表摘要
- 后端仍使用现有登录态鉴权，不因引入 `page agent` 放宽访问控制

## 日志设计

为便于排障与效果验证，关键位置增加详细日志。

建议记录：

- 请求用户标识
- 页面类型
- 当前路由
- 问题长度
- 是否包含 `selectionText`
- 是否命中当前页直接回答
- 是否触发站内检索
- 检索结果数量
- 模型耗时
- 返回来源数量

日志要求：

- 不原样记录超长正文
- 不原样记录完整敏感草稿内容
- 以摘要和长度信息为主，兼顾排障与隐私控制

## 影响范围

预计涉及以下区域：

- `apps/web/src`：全站入口、页面上下文构造、问答面板
- `apps/api/src/modules`：页面问答接口与站内检索编排
- `apps/api/src/config/env.ts`：复用现有 `DeepSeek` 配置

## 验收标准

### 文章详情页

- 提问“这篇文章讲什么”时，回答明显基于当前文章内容
- 提问“这段话是什么意思”时，若存在 `selectionText`，优先解释选中文本

### 资讯列表页

- 提问“当前页主要是什么内容”时，回答能说明当前筛选条件与当前结果特征
- 提问“有没有相关文章”时，能返回站内文章链接

### 订阅页

- 提问“每日、每周、即时有什么区别”时，能结合当前页面说明订阅含义

### 内容中枢页

- 提问“自动提取和 AI 提炼有什么区别”时，能结合页面现有功能说明
- 提问“系统里有没有类似文章”时，可触发站内检索并返回链接

### 通用

- 回答不得编造站内不存在的文章标题、链接或页面状态
- 检索型回答至少返回一条可用站内链接，若无结果则明确说明
- 日志可追踪一次请求是否走了检索链路

## 分阶段落地顺序

1. 定义统一 `page context` 结构
2. 新增后端页面问答业务接口
3. 接入文章详情页
4. 接入资讯列表页
5. 接入订阅页与内容中枢页
6. 补充日志与验证脚本

## 结论

第一阶段 `page agent` 最适合落地为“页面上下文问答助手”。

该方案具备以下特点：

- 满足“结合当前页面内容回答用户问题”的核心要求
- 满足“非详情页可检索站内内容并返回原文链接”的扩展要求
- 保持最小实现范围，不引入自动操作风险
- 可在后续阶段平滑扩展为“问答 + 导航建议”能力
