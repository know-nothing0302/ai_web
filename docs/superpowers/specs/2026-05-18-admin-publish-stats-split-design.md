# `ai_web` 内容发布与统计信息拆分设计

## 目标

本次设计聚焦于将当前混合在同一页面中的“内容发布”和“统计信息”拆分为两个独立的管理员页面，以解决以下问题：

1. 将页面名称从“内容中枢”统一调整为“内容发布”
2. 新增独立的“统计信息”管理员页面
3. 保留现有内容发布与文章管理功能，不改变业务能力
4. 让统计信息展示更聚焦、更容易判断是否已采集到数据
5. 降低当前管理员页的加载成本，优化页面响应速度

## 范围

### 本次包含

- 将“内容中枢”统一更名为“内容发布”
- 将当前 `/admin` 页面拆分为“内容发布”和“统计信息”两个路由页面
- 顶部导航增加独立“统计信息”入口
- `/admin` 重定向到新的“内容发布”路由
- `内容发布` 页面移除统计请求与统计展示
- `统计信息` 页面承载统计概览、趋势、分布、排行
- `统计信息` 页面默认时间范围改为 `近7天`
- 新增轻量统计状态接口，展示最近采集时间与事件总数
- 保持两页仅对管理员开放

### 本次不包含

- 外部统计接口改造
- 新的统计事件类型
- 图表库引入
- 导出报表
- 普通用户查看统计信息
- 统计数据缓存层重构

## 背景

当前系统已经具备以下能力：

- 路由 `/admin` 对应单一管理员页面
- 该页面同时承载内容提取、AI 优化、文章发布、文章管理、统计看板
- 统计接口已包含：
  - `GET /api/stats/overview`
  - `GET /api/stats/trends`
  - `GET /api/stats/distributions`
  - `GET /api/stats/rankings`
- 页面访问与文章浏览事件已正常写入 `analytics_events`

当前存在的主要问题：

- “内容发布”和“统计信息”职责混杂，页面信息密度过高
- `/admin` 首屏会同时拉文章管理数据和多组统计数据，导致体感变慢
- 统计信息不独立，用户不容易快速判断统计是否正常工作
- “内容中枢”命名与当前实际用途不完全一致

数据库已确认存在统计事件，说明统计链路本身已经启动并在工作。当前问题重点不在“是否采集”，而在“页面职责拆分、展示清晰度、默认口径和性能”。

## 设计原则

- 结构先于微优化，先通过职责拆分降低页面负担
- 保留原有内容发布能力，不做无关重构
- 统计信息单独成页，仅加载统计相关数据
- 默认展示口径优先可观察性，减少“看起来没数据”的误判
- 仅对管理员开放，不扩大访问范围
- 复用现有统计接口，优先小改后端、大幅改善前端结构

## 总体方案

本次采用“独立路由拆分，顶部双入口”方案。

页面拆分为：

1. `内容发布`
2. `统计信息`

这样可以直接解决：

- 命名不准确的问题
- 管理员页首屏请求过多的问题
- 统计信息不聚焦的问题
- 用户无法快速判断统计是否生效的问题

## 路由与导航设计

### 路由调整

建议调整为以下结构：

- `/admin`
  - 作为兼容入口，重定向到 `/admin/publish`
- `/admin/publish`
  - 内容发布页面
- `/admin/stats`
  - 统计信息页面

### 导航调整

顶部导航建议调整为：

- `资讯发现`
- `智能订阅`
- `内容发布`
- `统计信息`

### 文案统一

以下文案统一从“内容中枢”改为“内容发布”：

- 顶部导航文案
- 页面标题文案
- 页面上下文标题
- 无权限提示文案

## 页面职责拆分

### 内容发布页

页面定位：

- 内容生产与管理入口

保留现有功能：

- 文章来源提取
- AI 摘要提炼
- AI 优化建议
- 发布为正式文章
- 保存为草稿
- 文章列表管理
- 批量发布、批量删稿、批量改状态

移除以下内容：

- 所有统计概览卡片
- 趋势表
- 分布区
- 排行区
- 统计时间范围切换逻辑
- 统计接口请求逻辑

### 统计信息页

页面定位：

- 管理员查看访问、浏览、发布、推送、反馈相关统计数据的专用页面

承载内容：

- 统计范围切换
- 概览卡片
- 最近采集时间提示
- 每日趋势
- 栏目分布
- 反馈类型分布
- 热门文章排行
- 热门栏目排行

## 权限设计

`内容发布` 和 `统计信息` 均继续保持管理员可访问。

具体要求：

- 普通登录用户不可访问 `/admin/publish`
- 普通登录用户不可访问 `/admin/stats`
- 前端继续沿用当前管理员 allowlist 逻辑
- 后端统计接口继续保留管理员权限控制

这样做的原因：

- 内容发布涉及文章生产与状态变更
- 统计信息包含发布量、推送量、反馈量、热门内容等运营数据
- 不适合开放给普通用户

## 统计信息页设计

### 默认时间范围

统计信息页默认展示：

- `近7天`

支持切换：

- `近7天`
- `近30天`
- `今天`

选择该默认值的原因：

- 更容易展示近期有效数据
- 能降低“今天没有数据就误以为统计未启动”的概率
- 比近 30 天更聚焦，比今天更稳定

### 页面结构

从上到下建议分为 5 个区域：

1. 页面头部
2. 状态提示区
3. 概览卡片区
4. 趋势与分布区
5. 排行区

### 页面头部

展示内容：

- 标题：`统计信息`
- 副标题：说明这是管理员查看站内运营统计的页面
- 时间范围切换按钮

### 状态提示区

展示内容：

- 当前统计范围
- 最近采集时间
- 可选显示事件总数

如果当前范围没有数据，显示：

- `当前时间范围暂无统计数据`

### 概览卡片区

建议展示：

- 页面浏览量
- 访问用户数
- 文章浏览量
- 文章发布量
- 推送总量
- 推送成功率
- 反馈提交量
- 启用订阅数

### 趋势与分布区

趋势区展示：

- 日期
- PV
- UV
- 文章浏览量

分布区展示：

- 栏目浏览分布
- 栏目发布分布
- 栏目推送分布
- 反馈类型分布

展示方式继续使用：

- 卡片
- 表格
- 简单列表

不引入图表库。

### 排行区

展示：

- 热门文章 Top N
- 热门栏目 Top N

字段建议：

- 热门文章：
  - 标题
  - 栏目
  - 浏览量
- 热门栏目：
  - 栏目名称
  - 浏览量

## 后端接口设计

### 保留现有接口

继续复用：

- `GET /api/stats/overview`
- `GET /api/stats/trends`
- `GET /api/stats/distributions`
- `GET /api/stats/rankings`

### 新增接口

建议新增：

- `GET /api/stats/status`

用途：

- 用于统计信息页显示最近采集状态，帮助管理员快速判断统计是否正常写入

返回字段建议：

```json
{
  "latestEventAt": "2026-05-18T06:18:54.808Z",
  "totalEvents": 54,
  "todayEventCount": 7
}
```

字段说明：

- `latestEventAt`
  - 最近一条统计事件的写入时间
- `totalEvents`
  - 当前统计事件总数
- `todayEventCount`
  - 当日已写入事件数，用于判断当天是否有增量采集

### 鉴权要求

- `GET /api/stats/status` 与现有内部统计接口保持一致
- 仅管理员可访问
- 不开放给普通登录用户

## 性能优化设计

### 内容发布页优化

进入 `/admin/publish` 时，仅加载：

- 用户权限校验
- 栏目列表
- 文章列表

不再加载：

- `overview`
- `trends`
- `distributions`
- `rankings`
- `status`

这样可以减少首屏并发请求数量，并且发文、删文、批量操作后只刷新文章列表，不再顺带刷新统计数据。

### 统计信息页优化

进入 `/admin/stats` 时，仅加载统计相关接口：

- `overview`
- `trends`
- `distributions`
- `rankings`
- `status`

优化策略：

- 默认范围为 `近7天`
- 首次进入时并行拉取统计接口
- 切换时间范围时，仅刷新统计区
- 不加载文章发布区、文章管理区、AI 处理区状态

### 为什么这能解决“页面变慢”

当前变慢的主要原因，是 `/admin` 单页同时承担内容管理与统计看板：

- 页面状态多
- 首屏请求多
- 文章操作后附带统计刷新

拆分为两个页面后：

- 内容发布页更轻
- 统计页更聚焦
- 页面之间不再互相拖累

## 为什么当前会误以为“统计没启动”

根据已确认的数据情况，统计事件已经在写入数据库，说明统计链路正在工作。

当前误判的来源主要是：

- 内容页与统计页未真正拆分，观察点不清晰
- 默认展示不够聚焦
- 当前没有“最近采集时间”提示
- 页面整体变慢，导致用户主观感觉统计未刷新

新增独立统计页并显示 `latestEventAt` 后，可以让管理员直接区分：

- 统计未启动
- 当前时间范围暂无数据
- 统计已启动但页面尚未切换到合适范围

## 验证方式

### 功能验证

- 顶部导航显示：
  - `内容发布`
  - `统计信息`
- `/admin` 自动重定向到 `/admin/publish`
- `/admin/publish` 保留原有内容发布和文章管理功能
- `/admin/publish` 不显示任何统计区域
- `/admin/stats` 默认显示 `近7天`
- `/admin/stats` 能显示概览、趋势、分布、排行
- `/admin/stats` 能显示最近采集时间

### 统计有效性验证

- 打开文章详情页后，再进入 `/admin/stats`
- `latestEventAt` 有更新
- 文章浏览相关统计能反映新增数据

### 权限验证

- 非管理员访问 `/admin/publish` 被拒绝
- 非管理员访问 `/admin/stats` 被拒绝

### 性能验证

- `/admin/publish` 首屏请求量少于当前版本
- 发布、删除、批量操作后不再刷新统计接口
- `/admin/stats` 仅在进入页面和切换时间范围时请求统计接口

### 工程验证

- `apps/web` 构建通过
- `apps/api` 构建通过
- `apps/api` 类型检查通过
- 若新增 `stats/status` smoke 验证，则其结果通过

## 完成标准

完成后应满足：

1. “内容中枢”文案统一调整为“内容发布”
2. 新增独立管理员页面“统计信息”
3. `/admin` 重定向到 `/admin/publish`
4. `内容发布` 页面保留原功能且不加载统计
5. `统计信息` 页面默认展示 `近7天`
6. `统计信息` 页面显示最近采集时间
7. 管理员能明确判断统计是否正常采集
8. 页面响应速度较当前方案改善
9. 前后端构建与基础验证通过

## 结论

本次采用“内容发布与统计信息拆分为两个独立管理员页面”的方案。

该方案在不改变现有内容发布能力、不新增复杂统计基础设施的前提下，通过清晰的页面职责分离和轻量状态补充，直接解决当前的命名不准确、统计展示不清晰、管理员页加载偏重和“误以为统计未启动”的问题，是当前阶段最合适的最小可用结构优化方案。
