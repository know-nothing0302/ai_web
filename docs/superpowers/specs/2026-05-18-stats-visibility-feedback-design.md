# `ai_web` 统计信息可见性与反馈详情设计

## 目标

本次设计在现有“统计信息”页面基础上补充两个能力：

1. 普通用户不显示 `内容发布` 和 `统计信息` 管理入口
2. 在 `统计信息` 页面中展示最近反馈，并支持查看反馈详情

本次只做轻量增强，不新增单独的反馈管理标签页，不扩展复杂搜索和处理流转。

## 范围

### 本次包含

- 顶部导航按当前登录用户权限动态显示管理入口
- 普通用户隐藏：
  - `内容发布`
  - `统计信息`
- 普通用户直达 `/admin/publish` 或 `/admin/stats` 时继续显示无权限提示
- `统计信息` 页面新增 `用户反馈` 区块
- `用户反馈` 区块展示最近反馈列表
- 点击列表项后在当前页查看反馈详情
- 新增内部反馈读取接口，供统计页面调用

### 本次不包含

- 新增单独的 `反馈管理` 页面
- 反馈导出
- 反馈状态流转
- 复杂搜索、全文检索、多维筛选
- 外部反馈接口改造

## 背景

当前系统已经具备以下能力：

- `统计信息` 页面可以展示访问、浏览、发布、推送、反馈总量等统计数据
- 反馈数据已经通过 `/api/feedback` 成功写入数据库
- 反馈外部读取接口已经存在：
  - `GET /api/feedback/external`
- `统计信息` 页面当前已有前端 allowlist 限制
- 后端统计内部接口已经切换为内容发布 allowlist/operator 可读

当前仍存在两个体验问题：

1. 普通用户虽然无法真正读取统计数据，但顶部导航仍可能看到管理入口，不够收口
2. 统计页目前只有反馈总量和反馈类型分布，无法直接查看具体反馈内容

## 设计原则

- 管理入口的“可见性”与“可访问性”同时收口
- 反馈展示保持轻量，避免把统计页做成复杂后台
- 尽量复用现有反馈存储与分页能力
- 详情查看不跳新页面，减少操作打断
- 只补当前真正需要的信息，不做预支设计

## 可见性设计

### 目标用户

本次管理可见性继续沿用现有内容发布 allowlist/operator 规则。

也就是说，以下用户可以看到管理入口：

- 管理员角色用户
- 内容发布 allowlist 内的运营用户

以下用户不显示管理入口：

- 普通登录用户

### 导航策略

顶部导航改为根据当前登录用户动态生成：

- 所有用户都显示：
  - `资讯发现`
  - `智能订阅`
- 仅允许名单用户显示：
  - `内容发布`
  - `统计信息`

### 路由兜底

即使普通用户手动输入路由，仍保持现有兜底策略：

- 访问 `/admin/publish` 显示无权限提示
- 访问 `/admin/stats` 显示无权限提示

这样可以确保：

- 入口不暴露给普通用户
- 即使直达也不会越权

## 反馈展示方案

### 方案选择

本次采用“统计页内最近反馈列表 + 当前页弹层详情”的轻量方案。

原因：

- 用户反馈量通常不大
- 管理员查看反馈时更关注“最近发生了什么”
- 不需要增加额外路由和页面维护成本

### 页面结构

在现有 `统计信息` 页面底部新增一个独立区块：

- 区块标题：`用户反馈`
- 区块说明：用于查看最近提交的反馈内容

区块内部包含两部分：

1. 最近反馈列表
2. 反馈详情弹层

### 最近反馈列表

默认展示最近 `10` 条反馈。

列表字段建议为：

- 提交时间
- 反馈类型
- 页面标题
- 内容摘要
- `查看详情` 操作

展示说明：

- 内容摘要最多显示 1 到 2 行
- 页面标题优先展示 `pageTitle`
- 如果 `pageTitle` 缺失，可回退显示 `pageRoute`

### 反馈详情

点击 `查看详情` 后，在当前页打开轻量弹层，不跳转新页面。

详情展示字段：

- 反馈类型
- 完整反馈内容
- 联系方式
- 页面标题
- 页面路由
- 提交时间
- 用户 ID

弹层只读，不提供编辑或处理按钮。

## 接口设计

### 新增内部接口

新增内部接口：

- `GET /api/feedback/admin`

用途：

- 提供统计页中的最近反馈列表数据

### 鉴权

该接口使用与 `统计信息` 页面一致的内部读取权限：

- 管理员角色用户
- 内容发布 allowlist/operator 用户

普通用户不可访问。

### 查询参数

保持轻量分页查询能力：

- `startAt`
- `endAt`
- `type`
- `page`
- `pageSize`

统计页初版只需传：

- 当前时间范围对应的 `startAt`
- 当前时间范围对应的 `endAt`
- `page=1`
- `pageSize=10`

### 返回结构

返回结构沿用现有反馈列表模式：

- `items`
- `pagination`

其中每个反馈项应包含统计页详情所需字段：

- `id`
- `userId`
- `type`
- `content`
- `contact`
- `pageRoute`
- `pageTitle`
- `source`
- `createdAt`

本次不单独新增 `GET /api/feedback/admin/:id`，因为列表量小，详情可直接使用列表项完整数据。

## 前端交互设计

### 数据加载

`统计信息` 页面在加载原有统计数据时，同时请求最近反馈列表。

默认策略：

- 与统计范围联动
- 时间范围切换时重新拉取最近反馈

### 空状态

如果当前统计范围内没有反馈：

- 显示 `当前时间范围暂无反馈记录`

### 失败提示

如果反馈列表加载失败：

- 不影响已有统计卡片和趋势展示
- 在反馈区块内单独显示 `反馈加载失败，请稍后重试`

### 详情交互

点击 `查看详情` 后：

- 打开页内弹层
- 展示完整内容
- 可点击关闭返回列表

不增加额外路由状态，不做浏览器地址同步。

## 性能与复杂度控制

本次设计保持轻量：

- 只查询最近 10 条反馈
- 只做第一页，不在首版提供翻页器
- 不引入新状态管理
- 不新增独立详情接口
- 不增加图表或富交互组件

这样可以把改动控制在：

- 一个内部反馈读取接口
- 一组前端 API 方法
- 统计页一个反馈列表区块
- 一个轻量详情弹层

## 验证方式

完成后需验证以下内容：

1. 普通用户登录后顶部导航不显示 `内容发布` 和 `统计信息`
2. 普通用户手动访问 `/admin/publish` 或 `/admin/stats` 仍看到无权限提示
3. allowlist 用户仍可看到两个管理入口
4. `统计信息` 页面可正常显示最近反馈列表
5. 点击 `查看详情` 可看到完整反馈内容
6. 切换时间范围后，反馈列表会随之刷新
7. 反馈区块加载失败时，只影响反馈区块，不影响其他统计模块

## 完成标准

满足以下条件即可认为本次需求完成：

- 普通用户不可见管理入口
- 普通用户不可读取管理页面内容
- 统计页可以直接查看最近反馈
- 管理员或 allowlist 用户可以在当前页查看反馈详情
- 实现保持轻量，没有扩展出新的独立反馈后台
